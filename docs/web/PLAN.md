# Web Frontend Implementation Plan

> Implementation lives at [`web/`](../../web/). Spec at [`docs/web/ARCHITECTURE.md`](./ARCHITECTURE.md).

## Context

The acoustics_lab daemon (`acousticsd`) is mature: REST at `/api/v1/*`, two binary-protobuf WebSocket streams (`/stream/audio`, `/stream/infer`), SSE for job events, async job model, revision-tracked workspaces, sliding-window head storage with atomic hot-swap. Backend serves no static files. `web/` is a clean slate.

The frontend spec defines eight modules across three tabs (Dashboard, Workspace, Converter) plus a persistent Health Badge and floating Tiny Dashboard. The job is to build a slim SPA that lets a single operator record/edit audio clips, fine-tune classification heads, hot-swap weights, and observe live inference — all locally, with zero-latency dataset playback and no backend changes.

Pacing is deliberate: ample time, step-by-step. We will not pursue testing/benchmarking/CI infrastructure ([ARCHITECTURE.md §E.5](./ARCHITECTURE.md)).

## Locked decisions

- **Stack**: SvelteKit (latest) + TypeScript + Vite + Tailwind CSS v4. `@sveltejs/adapter-static`. Vite dev proxy → 127.0.0.1:8787.
- **Rendering**: pure-client SPA. Root layout sets `prerender = true; ssr = false` so build emits a static shell that hydrates browser-only.
- **No SSR/Node runtime**: build artifact is a static `web/build/` deployable to any static host or reverse-proxied.
- **Components**: hand-rolled Tailwind primitives. **Extracted only after a pattern is copy-pasted twice in real features.** No headless lib, no kit.
- **History**: memory ring via `GET /api/v1/jobs` for live state + JSONL backfill via `GET /api/v1/workspace/{id}/assets/training_logs/<job>.jsonl?after_seq=&limit=` for durability.
- **Converter workspaces**: marker tag `__converter__`, auto-name `converter-<uuid8>`. Client-side filter uses **name-prefix `converter-`** (not tag membership, since the daemon's list response omits `tags` — see corrections below). Tag is still set on creation as forward-compat metadata.
- **No tests, no benchmarks, no CI**.

  - **Eager-tier auto-refill after delete** (2026-05-19, same-day follow-up): `deleteHistoryEntry` now calls `refillEagerAfterDelete(workspaceId)` after `removeFromHistory` lands. The motivating bug: on a fresh mount, deleting an eager row left the "Show N older runs" badge visually unchanged (the operator deleted one entry but the count stayed at the same N). Root cause: `removeFromHistory` drops the jobId from BOTH `historyByWs` and `discoveredByWs`, so the `loadable = discovered − loaded` math stayed numerically identical while the visible eager row count silently went from `INITIAL_VISIBLE` to `INITIAL_VISIBLE − 1`. The refill calls `loadBatch(workspaceId, gap)` for the exact eager-tier shortfall whenever there are discovered-but-not-loaded entries remaining; the just-pulled entry leaves the loadable pool so the badge ticks down by one as the operator expects, AND the eager tier returns to its budget. No-op when `historyByWs.length >= INITIAL_VISIBLE` (the older-expanded path absorbs the gap via the slice shift natively) or when the discovery pool is exhausted (nothing to fetch). Fire-and-forget so the primary delete resolves promptly; a transient fetch failure leaves the eager tier short until the next mount / older-expand refresh, which is the right recovery layer for a background polish-the-list call.
  - **Eager-tier skeleton placeholder during auto-refill** (2026-05-19, follow-up): with the auto-refill in place, the eager tier still visibly shrunk for the duration of the backfill fetch and re-grew when the new card mounted — a "shrink → re-grow" judder the operator noticed every time they deleted from a freshly-mounted page. Fix: brackets the in-flight `loadBatch` await with a refcount on `autoRefillingByWs: SvelteMap<Uuid, number>` (a counter, not a boolean, so two rapid overlapping refills both keep the flag true until BOTH settle), exposed via `autoRefillingFor(ws)` + a unified `eagerSkeletonCountFor(ws)` that returns the exact `INITIAL_VISIBLE − history.length` gap whenever hydration OR auto-refill is in flight. `TrainHistory` reads the count and renders that many `<li>` skeletons via a single `{#each Array.from({length}, (_, i) => i) as i}` loop right after the eager rows; this replaced the previous hard-coded `showEagerSkeletons` flag + two duplicated `<li>` blocks. Skeleton shape (`h-10` collapsed-card footprint + dashed-pulse glyph strip) matches `TrainHistoryItem`'s resting height byte-for-byte so the eager `<ul>` keeps the same total height through the delete → drain → backfill transition; the operator sees a single in-place card swap rather than a shrink + re-grow. Same skeleton primitive now serves both the initial hydration path (history empty, `gap = INITIAL_VISIBLE`) and the post-delete refill path (history short by one, `gap = 1`).
  - **Older-tier placeholder count matches the click-time batch size** (2026-05-19, follow-up): clicking "Show N older runs" used to paint a single 40 px skeleton regardless of N — operator clicks "Show 3 older runs", sees 1 placeholder, then 3 entries pop in (with the bonus indignity of a 40 px → 3 × 40 px layout jump as the placeholder collapses and the entries' `<ul>` expands). Click "Show 3 older runs" → 3 skeletons appear → 3 rows replace them; click "Show 8 older runs" → 5 skeletons appear → 5 rows replace them and a "Load 3 more" affordance surfaces below. Snapshot semantics: never paint MORE skeletons than rows will land (the visible badge → placeholder count → rendered row count chain is honest at click time, even if a backend race prunes one of the discovered entries during the fetch); a `pending > 0` guard skips the slot entirely when the in-flight load resolves to a no-op (`older.length >= PAGE_SIZE` or `loadable === 0`).
- **Slice D.1 — Head export (`.alpkg`)** (2026-05-18) — **pure-client export pipeline for trained heads, surfaced from the Deploy module's `HeadsTable`. Import is deferred (see [Sub-slice strategy for Slice D](#sub-slice-strategy-for-slice-d) below).**
  - **No daemon changes required.** The asset GET surface (`modules/api/routes/dataset.rs:get_asset`) already resolves `/workspace/{ws}/assets/heads/<head_id>.{mpk,json}` -- only `.tmp/` is blocked for reads, and the daemon writes the per-head `<head_id>.{mpk,json}` files at publish time. The `PUT /assets/{*path}` upload allowlist (`datasets/` + `converters/` only) is irrelevant for export. Both halves of the head export pull from the assets API for symmetric path semantics.
  - **`.alpkg` container** ([utils/alpkg.ts](../../web/src/lib/utils/alpkg.ts)): hand-rolled ZIP encoder (~200 LOC) over the browser-native `CompressionStream('deflate-raw')`. Three entries per archive -- `package.json` at root + `head/<head_id>.mpk` + `head/<head_id>.json` under a single `head/` subdirectory -- so an operator can open the `.alpkg` in any unzip tool and the importer can read both halves directly. CRC-32 table-based; fixed DOS-epoch timestamps (1980-01-01) for deterministic byte output across re-exports of the same head; ZIP32 4 GiB ceiling enforced as a typed `RangeError` even though trained heads sit in the low-MiB range. `safeFilenameSlug()` reduces the workspace name to the `AssetPath` allowlist (`[A-Za-z0-9._-]`) so the SaveAs default reads as `<ws-slug>-head-<id8>.alpkg`.
  - **alpkg manifest schema** is a minimal three-field envelope: `format: 'alpkg'` + `version: 1` + `exported_at: <RFC3339>`. Everything else is derivable from the archive itself -- the `.alpkg` extension + the `head/<id>.{mpk,json}` tree name the payload kind, and the embedded `head/<id>.json` (the daemon's authoritative `HeadManifest`) carries `head_id`, `workspace_id`, `workspace_revision`, `sha256`, `n_classes`, `size_bytes`, `labels`, and `created_at`. The earlier draft of this entry duplicated all of those fields inside `package.json`; that was dropped on review because the duplication invited drift between the envelope and the daemon-authored payload while adding nothing the importer can't verify by hashing the embedded `.mpk` against `manifest.sha256` from the embedded `.json`.
  - **Export orchestrator** ([api/heads-export.ts](../../web/src/lib/api/heads-export.ts)): five typed phases (`fetching-weights → fetching-manifest → validating → packing → downloading`) emitted via an optional `onprogress` callback so the UI can morph copy. Validation order matches the spec's "metadata first, then size + hash for the weight data": parse + structural-check the manifest, cross-check against the `HeadRecord` the operator clicked (a race where the row went stale between list + click surfaces here rather than packing a stale artefact), then verify the weight bytes' length + SHA-256 against the manifest. Pipeline-failure is the typed `ExportError { phase, message, cause? }` so the surface picks the right copy without parsing the message string. Re-serialises the parsed `HeadManifest` with a deterministic 2-space indent before packing so the alpkg's `files.manifest.sha256` decouples from whatever whitespace the daemon's serde-json emitter happens to produce.
  - **Endpoint additions** ([api/endpoints.ts](../../web/src/lib/api/endpoints.ts)): `heads.weightsAssetPath(ws, id)` + `heads.manifestAssetPath(ws, id)` are pure URL builders that route through the generic asset-GET surface (`/assets/heads/<id>.{mpk,json}`) rather than the dedicated `/heads/{id}` route. The dedicated route's orphan-index filter is redundant here because the export entry point is a row in the cached `HeadsTable` -- the row already implies the head is in the daemon's `heads.json` index.
  - **HeadManifest TS type fix** ([api/types.ts](../../web/src/lib/api/types.ts)): refactored `HeadRecord` + `HeadManifest` to share a `HeadCore` base so the manifest carries the daemon's actual `workspace_id` field (previously missing in TS, the `as unknown` workaround was the symptom) and drops the never-actually-served `status` derivation. List rows keep `status`; manifest reads keep `workspace_id` + `labels`. The two endpoints' wire shapes now agree with the typed models, which the export pipeline's `validateManifest` cross-checker relies on.

### Corrections to the spec (verified against the running daemon)

The original Slice B prose assumed wire shapes that don't match the actual API. Discovered by direct probing in [docs/API.md](../API.md) cross-referenced with [modules/api/routes/workspace.rs](../../modules/api/routes/workspace.rs); future sessions should treat these as authoritative:

1. **`GET /api/v1/workspace` (list)** returns only `{id, name, created_at}` per entry — **no `tags`, no `workspace_revision`, no `head_count`**. The `__converter__` filter therefore can't read tags from the list and uses the workspace-name prefix `converter-` instead.
2. **`GET /api/v1/workspace/{id}` (detail)** returns `{id, name, created_at, workspace_revision, heads[]}` — **no `tags` either**. Only `POST` / `PATCH` responses carry tags.
3. **Workspace delete concurrency**: the daemon's `JobRegistry` admits one `WorkspaceDelete` at a time globally. Firing N parallel `DELETE`s rejects N-1 with `409 conflict` (`fs: job conflict: conflicts with running job ... (WorkspaceDelete)`). Both the per-item and bulk delete flow through a single client-side `deleteChain` that serializes ACK + SSE-terminal before kicking off the next.
4. **Heads bundle**: heads come on the workspace detail. There IS a separate `GET /workspace/{id}/heads` endpoint that returns `{ heads: HeadManifest[] }` (with `labels` filled in, omitted from the inline `HeadRecord`); the original PLAN claim that this endpoint didn't exist was wrong — it does, and Slice C wires it via `heads.list` for the refresh-after-train flow.
5. **Training jobs are in the unified `/jobs` surface** (bridge landed 2026-05). The training producer's `try_acquire(JobType::Train, ...)` admission route + `JobHandle::append_log` event broadcast wire it into the cross-cutting `file_mgr::JobRegistry`. Frontend implication: live training progress streams over SSE at `GET /api/v1/jobs/{id}/events` (each `JobEvent.message` carries a JSON-stringified typed `TrainEvent`); the polled `GET /workspace/{id}/training/{job}` endpoint stays as the snapshot surface used by `recover()`'s discovery hop only. Earlier Slice C notes that described the asymmetry as in-place are now historical (see the 2026-05-15 update in the [Slice C entry](#shipped) above).

## Strategy: vertical slices

We **do not** build full horizontal layers (audio infra → IDB → stores → components → modules). Each layer's correctness only becomes visible when an actual feature wires it end-to-end; building all three before the first feature compounds undetected integration risk.

Instead: five vertical slices, each delivering working user value. Shared foundations are extracted only when a second slice needs them.

## Cross-cutting conventions

### Proto decoder

- Source-of-truth: [modules/proto/\*.proto](../../modules/proto). Never copied into `web/`.
- Implementation: hand-written decoder at [src/lib/stream/proto.ts](../../web/src/lib/stream/proto.ts) covering `Envelope`, `AudioFrame`, `InferenceFrame`, `TopK`. ~150 LOC, zero deps.
- Wire-format dispatch on field tag: receiver silently drops unknown tags per proto3 unknown-field semantics ([PROTO.md](../PROTO.md)).
- When `.proto` files change, the TS decoder must be updated in lockstep. The total field count is tiny enough that visual inspection suffices; consider adding back codegen only if the schema grows.

### Error envelope handling

- All API responses parsed for `{ error, code }`. Fetch wrapper rejects with a typed `ApiError { status, code, message }`.
- Stores translate `ApiError.code` → user-facing copy via [src/lib/utils/error-copy.ts](../../web/src/lib/utils/error-copy.ts).
- The one special case: SSE `event_gap` (409) is **not** an error — it triggers JSONL backfill in `subscribeJob`.

### Read-your-writes — mic policy

- `POST /api/v1/mic` returns new `version`. Immediately re-GET with `?min_version=<new>`, retry on 425 `too_early` with backoff (≤3 attempts), then surface success. Without this, the UI flickers back to the pre-write state.

### WS lifecycle

- Single shared Worker owns both `/stream/audio` and `/stream/infer`. Drains continuously regardless of UI visibility (backpressure threshold 64 frames = 1.28 s at 50 Hz — never let the renderer be the consumer).
- Renderer subscribes to Worker's ring via `postMessage` transferables. Audio frames decoded in Worker; main thread receives `Float32Array` PCM windows.
- On WS close (1011 lagged): full reconnect, accept frame loss (no replay). Show momentary "stream interrupted" indicator.

### Active head provenance

- Header strip displays `head_id[:8]@v{head_version}` from latest inference frame. Hover → popover with `source_workspace_id`, `workspace_revision`, `labels`.
- If `GET /api/v1/active` returns `source_workspace_alive: false`, header strip turns amber: "active head's source workspace deleted".

### TypeScript types match Rust

- All domain types in [src/lib/api/types.ts](../../web/src/lib/api/types.ts). Use discriminated unions where Rust uses tagged enums (e.g., `ActiveResp` with `origin: 'head' | 'default'` discriminates on whether `source_*` fields exist).
- Keep `as const` enums for codes/states. No string literals scattered in feature code.

### Vite proxy

```ts
server: {
  proxy: {
    '/api':           { target: 'http://127.0.0.1:8787' },
    '/stream/audio':  { target: 'ws://127.0.0.1:8787', ws: true },
    '/stream/infer':  { target: 'ws://127.0.0.1:8787', ws: true },
  }
}
```

Forgetting `ws: true` is silent — upgrade fails and debugging takes an hour. Lock this in Slice A.

### `__converter__` tag convention

- Set on workspace creation in the converter wizard. Filtered out of Workspace Tab's list (`tags.includes('__converter__')` excludes). Shown in Converter Tab.
- Backend doesn't know about the marker; it's purely a frontend convention.
